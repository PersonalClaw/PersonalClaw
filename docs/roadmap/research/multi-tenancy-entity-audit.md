# Multi-Tenancy Entity Audit — Provider Seams, Single-Tenant Assumptions, and Harness-Side Readiness

**Status:** RESEARCH (code-evidence audit, 2026-07-14; §6-7 rescoped 2026-07-14 to harness-side readiness)
**Scope:** Audit PersonalClaw's readiness to be a good citizen of multi-tenant providers — a task provider whose tasks aren't all assigned to this harness's owner, a trigger store containing other users' triggers, a memory provider returning other contributors' memories. PersonalClaw itself stays personal; the harness handles foreign-attributed records sensibly, and sharing semantics live at the shared-store / application level.
**Method:** direct code reads of every provider seam and entity store in `src/personalclaw/`, plus overlap mapping against the 27 roadmap plans. Every claim carries a file anchor. Paths relative to `PersonalClaw/src/personalclaw/` unless noted.

---

## 0. Executive Summary

PersonalClaw's provider architecture is genuinely strong on the *capability* axis — models, channels, agents, search, actions, prompts, skills, tools all plug in via apps with zero core edits (`docs/architecture/provider-boundary.md`). But the *entity-storage* axis is almost uniformly local-single-file: schedules, projects, inbox items, knowledge items, memory records, chat sessions, and notifications all persist as JSON/JSONL/sqlite under one un-keyed `~/.personalclaw/` home (`config/loader.py:161 config_dir()`). Exactly **one** entity today has a true storage-provider seam a remote backend could implement: **tasks** (`tasks/provider.py TaskProvider` ABC + `tasks/registry.py` aggregation with per-provider routing and `readonly` write-guarding). Memory has a *contract* (`memory_providers/base.py MemoryProvider`) shaped right but a registry that hard-defaults to the native store and a resolution path that bypasses the registry entirely.

Identity is the deeper gap: there is **one owner** everywhere. `GatewayServices.owner_id` (gateway_services.py:58) is "the primary owner's channel-user id"; auth modes (`auth/modes.py`) authenticate *the machine's user*, not *a* user; the SEL logs `caller_identity` as a session key (sel.py:68), not a principal; approvals route to "the" dashboard/DM. No entity record carries an author/actor field. Two usernames' records flowing through one harness today would corrupt nothing (append-mostly stores) but would be *indistinguishable* — no attribution anywhere.

The good news: the harness-side fix is small. What the harness needs is a username identity, contributor attribution on entity records, an owner-filter at trigger arm time, and provenance-weighting on recall (§6) — sharing semantics, permissions, and coordination belong to the shared store / application design, not the harness.

## 1. Provider-Pluggable Entities Today (the seam inventory)

The extension registry is `providers/registry.py`: apps declare `provider: {type, implementation}` in their manifest (`apps/manifest.py:453 PROVIDER_TYPES` = {model, agent, task, channel, inbox, skills, knowledge, memory, notification, tool, workflow, search, action, prompt} — guarded by `test_manifest_types_match_handlers` to equal the runtime `_TypeHandler` set). `providers/loader.py` loads each enabled app and registers contributions through typed handlers. Apps are separate OS processes with their own network stacks (`apps/permissions.py`: `network` is declaration-only, unenforced by design) — **an app can already host or call a remote backend today**; what varies is whether the *entity contract* lets that backend own the data.

## 3. Single-Tenant Assumptions (what breaks first)

1. **One owner identity, stringly and channel-scoped.** `GatewayServices.owner_id` — "the primary owner's channel-user id, or `""` if unset" (gateway_services.py:58); `DashboardState(owner_id="")` (state.py:663). The Slack app *auto-claims* the first DM sender as owner (trust-on-first-use: `apps/slack-channel/slack_runtime/events.py:1325-1338 claim_owner`; handler.py:759), persisted via the credential store ("runtime-discovered credentials such as a channel app's auto-claimed owner id", loader.py:240). events.py:604 says it outright: "Multi-user is still [unsupported]". There is no `Principal`/`Actor` type anywhere in core.
2. **Auth authenticates access, not identity.** `auth/modes.py`: none / local_token / api_key / oauth2. Tokens gate *the whole gateway*; `generate_token(user_id, app=...)` (dashboard/token_auth.py) carries an app claim for the permission sandbox but the `user_id` is the single dev/owner. EXTERNAL-ACCESS deliberately scopes itself to "one user letting their own tools talk to their own assistant — not a multi-tenant API product" (its §Overview soul guardrail), and that scope stands.
3. **One un-keyed home.** Everything under `config_dir()` (`~/.personalclaw` or `PERSONALCLAW_HOME`, loader.py:161) with no tenant/user key in any path. Two humans on one backend is fine (homes are per-human); the assumption that bites is **entity ids are locally minted with no origin** — `t-<hex8>`, `p-<hex8>`, trigger ids: 8-hex collision space with no `machine_id`/`origin` field, so merging two harnesses' stores can collide and can't attribute. (DURABILITY-AND-SYNC already introduces a stable per-machine `machine_id` in its shard manifest — the natural origin key.)
4. **No author on anything.** `Task` has no creator/assignee; `ScheduleJob` has `created_by: agent|user|system` (a *kind*, not a *who*); memory records have no author; `TaskComment.add_comment(author="")` accepts a free string nobody sets. First thing a shared backend needs and the cheapest to add.
5. **Approvals route to "the" user.** Tool approval prompts, trust/YOLO state (`trust_mode.py` — ONE process-global auto-approve state), needs-input inbox, `DashboardState.notify` — all assume the one dashboard/DM. Correct for a single-owner harness; WORKFLOWS-V2's owner binding ("only the run's requester can approve from a shared channel; non-owner replies ignored") already covers the shared-channel case.
6. **SEL identity is a session key.** `sel.py:68 caller_identity: str  # session key or user identifier` — fine for a single-user harness; noted as an observation only.
7. **sync_bridge.py is a proto-multi-tenant surface, narrowly.** It hands a dashboard session to a channel thread (`handoff_to_channel` → `open_dm(owner_id)` → `set_channel_link`) and back (`!resume`). The session becomes visible in a *shared* medium (a Slack channel), and the owner-binding rule exists precisely because non-owners can now see and reply. But linkage is owner-centric (DM to owner_id); nothing models a second participant. It's evidence the channel layer is where second identities first appear — sender ids already flow through `ChannelMessage`, and EXTERNAL-ACCESS §6's `sender_trust.json` (per-transport `{sender_id: {state, display, paired_at}}` + pairing codes) is the first core store keyed by *someone other than the owner*.
8. **App platform is tenant-ready-ish where it matters.** Per-app DATA_DIR (`storage` permission), app-scoped 1-hour tokens with cookie/Authorization stripping at the proxy (apps.py::api_app_proxy), permission middleware in all auth modes. An app CAN host a team-backend client today (own process, own network). What's missing is not app capability — it's the entity contracts (§1/§2) and identity fields the app would need to speak.

**What breaks first if two humans share an entity backend:** nothing crashes — attribution silently collapses. Both harnesses write as "the user"; approval prompts land on whichever dashboard is attached; id collisions are possible; deletes resurrect on the other side (no tombstones — DURABILITY-AND-SYNC adds them); and any shared *automation* would fire N times (once per harness) because nothing filters arming down to the owner's own triggers.

## 6. Harness-Side Readiness Gaps

What the harness itself is missing to behave correctly as one client of a multi-tenant store (everything else — sharing semantics, permissions, coordination — is the shared store's / application's design space, not the harness's):

1. **No username identity.** One `owner_id` string (gateway_services.py:58), TOFU-claimed by a channel app; onboarding collects only a display name (`dashboard.user_name`). The harness has no stable identity string to attribute its own writes with.
2. **No contributor attribution on entity records.** `Task` has no author/assignee; `TaskComment.author` is an unset free string; memory records carry no contributor; `ScheduleJob.created_by` is a *kind* (agent|user|system), not a *who*. A provider returning records attributed to other people has nowhere to put the "who," and the harness would treat everything as the owner's.
3. **No owner-filter on triggers.** Nothing distinguishes "created by this harness's owner" from "created by someone else" — a trigger source containing other users' rows would have them all armed and fired locally. The filter belongs at arm time in the (post-substrate) `TriggerService`.
4. **No provenance-weighting on recall.** Memory recall has no contributor labels and no owner-preference in ranking; foreign-contributed memories would rank and render indistinguishably from the owner's own. (The labeled-provenance rendering for cross-project recall — WORK-R14 — is the existing mechanism to reuse.)

---

## 7. Recommended Slot-In

These gaps are closed by the rescoped plan `docs/roadmap/plans/TEAM-SHARED-ENTITIES.md` ("Multi-Tenant Entity Readiness — The Harness as a Good Citizen of Shared Stores"): a first-boot username + optional attribution fields defaulting to the owner, per-entity tolerance semantics (assignee-aware tasks with MINE-only work counters; owner-only trigger arming with foreign rows visible-but-inert; contributor-labeled, owner-weighted memory recall), the trigger-store provider seam (gated on AUTOMATION-SUBSTRATE's `triggers.json` unification — one seam, not four), and one proof-of-concept trigger-provider app validating the integration path end-to-end. Where schemas are still forming (triggers.json, v2 ledger rows), optional author/contributor string fields are reserved at design time.
